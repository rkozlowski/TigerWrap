# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

TigerWrap is a schema-first code generator for SQL Server: it produces strongly-typed C# wrappers for stored procedures and enum tables (no ORM). See `AGENTS.md` for contribution conventions (coding style, commit/PR format, SQL file naming); this file focuses on the parts of the architecture you can't discover from a single file.

## Build, test & run

- `dotnet build ItTiger.TigerWrap.Core\ItTiger.TigerWrap.Core.csproj` / `dotnet build ItTiger.TigerWrap.Cli\ItTiger.TigerWrap.Cli.csproj` — build the C# projects. The output assembly of the CLI is named `tiger-wrap`.
- `dotnet test ItTiger.TigerWrap.Tests\ItTiger.TigerWrap.Tests.csproj` — run the tests (xUnit v3). Tests marked `Trait("Category", "RequiresSqlServer")` exercise a local SQL Server (server `.`, integrated security) with disposable databases and skip themselves when no server is available.
- `dotnet run --project ItTiger.TigerWrap.Cli -- --help` — run the CLI locally (args after `--`).
- The SSDT project `TigerWrapDb.sqlproj` does **not** build with `dotnet build`; use Visual Studio MSBuild to validate SQL changes without a live server, e.g. `& "C:\Program Files\Microsoft Visual Studio\18\Community\MSBuild\Current\Bin\MSBuild.exe" TigerWrapDb\TigerWrapDb.sqlproj /p:Configuration=Debug`.
- `dotnet build ItTiger.TigerWrap.Installer -c Release` — build the Inno Setup installer (runs `BuildInstaller.ps1`; requires Inno Setup 6).

Targets `net10.0`, nullable + implicit usings enabled. Version comes from `Version.props` (imported by the C# csproj files); currently `0.9.1`.

## The one thing to understand first: the generator runs in SQL, not in C#

The actual code-generation logic does **not** live in this repository's C#. It lives in the **`TigerWrapDb` metadata database**, in the `[Toolkit]` schema stored procedures (`TigerWrapDb/Stored Procedures/`). The C# side is a thin orchestration + I/O layer:

1. `GenerateCodeCommand` resolves a connection and project, then calls `ToolkitDbHelper.GenerateCodeAsync`, which executes `[Toolkit].[GenerateCode]`.
2. That SP returns **rows of generated code text** (`GenerateCodeResult`: `Id`, `CodePartId`, `Schema`, `Text`).
3. The C# command groups those rows by `CodePartId` (Enums / ResultTypes / TvpTypes / SpWrappers / CodeHeader / CodeBootstrap / CodeEnd) and `Schema`, then concatenates and writes them to files according to `--output-type` (SingleFile / SplitPerType / SplitPerSchema / SplitPerSchemaAndType).

So: to change *what code is generated*, edit the SQL SPs in `TigerWrapDb`. The C# only decides *how the returned text is split into files* and handles CLI/UX. Project configuration (which SPs/enums to wrap, naming normalization, language options) is stored as data in `TigerWrapDb` and manipulated through `projects` subcommands — not in local config files.

`TigerWrapDb/DeploymentScripts/*.sql` (full deploy + version upgrade) are **SSDT-generated release artifacts** — they are not regenerated by editing source SQL and need an SSDT publish/script step at release time. `ExpectedDbInfo` in Core must track `TigerWrapDb/Scripts/Script.Version.sql` (schema version + API levels).

## Self-hosting: `ToolkitDbHelper` is itself generated — do not hand-edit

`ItTiger.TigerWrap.Core/ToolkitDbHelper.*.cs` (Bootstrap, Enum.Enums, Toolkit.ResultTypes, Toolkit.StoredProcedures) are **TigerWrap output** — the tool wraps its own `[Toolkit]` schema (note the `// <auto-generated>` headers, project `TigerWrapToolkit`). These partial-class files are regenerated by running TigerWrap against `TigerWrapDb`; any manual edits are lost on regeneration. If a wrapper method or `GenerateCodeResult`/enum shape needs to change, change the SP/metadata in `TigerWrapDb` and regenerate, don't patch the C#.

Hand-written Core code lives in the non-generated files: `ToolkitHelper.cs` (connection-store factory, language/options resolution helpers), `DbInfoValidator.cs` / `ExpectedDbInfo.cs`, `Enums.cs`, `ProjectInfo.cs`.

## CLI structure (ItTiger.TigerCli)

The CLI is built on the **TigerCli** application framework (`ItTiger.TigerCli` NuGet package) with **TigerQuery** for SQL Server connectivity and script execution (`ItTiger.TigerQuery`, `.Core`, `.CliCore`). `TigerWrapApp.cs` composes the app with `TigerCliApp.CreateBuilder()` and registers the command tree:

- `connections` (list/show/add/edit/delete) — provided by `SqlServerConnectionCommands.Configure` from `ItTiger.TigerQuery.CliCore`, backed by the shared connection store
- `db` (info/upgrade) — inspect a TigerWrap database (`[Toolkit].[GetDbInfo]`) and run the supported schema upgrade via the TigerQuery engine in SqlCmdEx mode
- `projects` (list/show/add/update) with sub-branches `sp` (add/remove), `enum` (add/remove), `norm` (add/remove — name normalization rules)
- `generate-code`, `languages-list`

Conventions to follow when adding commands:
- Commands live under `Commands/<Feature>/`, one class per verb, deriving from `TigerCliAsyncCommandHandler<TSettings>`; settings classes use `[TigerCliArgument]`/`[TigerCliOption]` attributes (prompting, providers, validation are declarative).
- Selection lists come from **providers** registered in `TigerWrapApp` (`connections`, `projects`, `schemas`, `languages`, ...); reference them via `Provider = "name"` on the attribute. The default prompt mode is `Yes` (every missing option is prompted unless `Promptable = TigerCliPromptable.No`); interactive UI (`TigerTui` prompts, `RunActivityAsync` progress) automatically degrades in `--non-interactive` mode.
- Most DB-touching commands start with `ToolkitHelper.TryResolveDbHelperAsync(connectionStore, name)`, which resolves the connection, builds a `ToolkitDbHelper`, and runs `DbInfoValidator.ValidateAsync` (checks the logical DB name and API-level compatibility via `ExpectedDbInfo`). The `db` commands intentionally bypass this validation (they must work against not-yet-upgraded databases) and use `[Toolkit].[GetDbInfo]` directly.
- Exit codes are `ToolkitDbHelper.ToolkitResponseCode` values (SP return values cast directly into this enum; CLI-side codes are 1000+, TigerCli-mapped codes 2002+). The app maps them via `UseExitCodes<ToolkitResponseCode>` in `TigerWrapApp`.
- Render output with `TigerConsole` + `CliTable`/`CliList`/`CliDetails`; localizable text goes through `settings.T(...)`/`settings.E(...)`.

## Connections & secrets

Connections are managed by TigerQuery's `SqlServerConnectionStore` (created in `ToolkitHelper.CreateDefaultConnectionStore`), persisted to `connections.json` under `%AppData%\ItTiger.net\TigerWrap` (Windows) or `~/.config/ItTiger.net/TigerWrap` (other). Passwords are DPAPI-encrypted (`CurrentUser` scope) via TigerQuery's password protector, so stored SQL passwords are Windows- and user-specific. These connections point at the TigerWrap **metadata** DB, not the application database whose procedures are being wrapped (the target DB is chosen per project / via `--database-name`).

## Repo layout

- `ItTiger.TigerWrap.Core/` — connection handling, validation, models, and the generated `ToolkitDbHelper` wrappers (Dapper + `Microsoft.Data.SqlClient`).
- `ItTiger.TigerWrap.Cli/` — the `tiger-wrap` CLI (TigerCli/TigerQuery, embedded `Resources.resx`).
- `ItTiger.TigerWrap.Tests/` — xUnit v3 tests: app/registration tests via `TigerCliAppTestHost`, plus live SQL Server tests (`Category=RequiresSqlServer`).
- `TigerWrapDb/` — SSDT SQL project; the generator engine. SQL grouped by object type (`Tables/`, `Stored Procedures/`, `Functions/`, `Security/`, `DeploymentScripts/`).
- `ItTiger.TigerWrap.Installer/` — Inno Setup installer (`Installer.iss`, `BuildInstaller.ps1`); packages the CLI into `{app}\cli` and the deployment scripts into `{app}\sql`.
- `docs/` — user docs (`CLI.md`, `ENUMS.md`, `WRAPPERS.md`, `INSTALL.md`).
