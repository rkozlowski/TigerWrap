# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

TigerWrap is a schema-first code generator for SQL Server: it produces strongly-typed C# wrappers for stored procedures and enum tables (no ORM). See `AGENTS.md` for contribution conventions (coding style, commit/PR format, SQL file naming); this file focuses on the parts of the architecture you can't discover from a single file.

## Build & Run

- `dotnet build TigerWrap.sln` — build everything. The SQL project (`TigerWrapDb`) needs SSDT / Visual Studio SQL tooling; on machines without it, build the individual C# projects instead.
- `dotnet build ItTiger.TigerWrap.Cli/ItTiger.TigerWrap.Cli.csproj` — build just the CLI. The output assembly is named `tiger-wrap`.
- `dotnet run --project ItTiger.TigerWrap.Cli -- --help` — run the CLI locally (args after `--`).
- `dotnet build ItTiger.TigerWrap.Installer -c Release` — build the installer (runs `BuildInstaller.ps1`).

Targets `net10.0`, nullable + implicit usings enabled. Version comes from `Version.props` (imported by both C# csproj files); currently `0.9.0`.

There is **no test project** in the tree. Validate changes by building and by exercising the CLI against a real TigerWrap database.

## The one thing to understand first: the generator runs in SQL, not in C#

The actual code-generation logic does **not** live in this repository's C#. It lives in the **`TigerWrapDb` metadata database**, in the `[Toolkit]` schema stored procedures (`TigerWrapDb/Stored Procedures/`). The C# side is a thin orchestration + I/O layer:

1. `GenerateCodeCommand` resolves a connection and project, then calls `ToolkitDbHelper.GenerateCodeAsync`, which executes `[Toolkit].[GenerateCode]`.
2. That SP returns **rows of generated code text** (`GenerateCodeResult`: `Id`, `CodePartId`, `Schema`, `Text`).
3. The C# command groups those rows by `CodePartId` (Enums / ResultTypes / TvpTypes / SpWrappers / CodeHeader / CodeBootstrap / CodeEnd) and `Schema`, then concatenates and writes them to files according to `--output-type` (SingleFile / SplitPerType / SplitPerSchema / SplitPerSchemaAndType).

So: to change *what code is generated*, edit the SQL SPs in `TigerWrapDb`. The C# only decides *how the returned text is split into files* and handles CLI/UX. Project configuration (which SPs/enums to wrap, naming normalization, language options) is stored as data in `TigerWrapDb` and manipulated through `projects`/`connections` subcommands — not in local config files.

## Self-hosting: `ToolkitDbHelper` is itself generated — do not hand-edit

`ItTiger.TigerWrap.Core/ToolkitDbHelper.*.cs` (Bootstrap, Enum.Enums, Toolkit.ResultTypes, Toolkit.StoredProcedures) are **TigerWrap output** — the tool wraps its own `[Toolkit]` schema. Note the `// <auto-generated>` headers ("Tool version 0.8.7", project `TigerWrapToolkit`). These partial-class files are regenerated by running TigerWrap against `TigerWrapDb`; any manual edits are lost on regeneration. If a wrapper method or `GenerateCodeResult`/enum shape needs to change, change the SP/metadata in `TigerWrapDb` and regenerate, don't patch the C#.

Hand-written Core code lives in the non-generated files: `ToolkitHelper.cs` (language/options resolution helpers), `DbInfoValidator.cs` / `ExpectedDbInfo.cs`, `Services/ConnectionService.cs`, `Models/`.

## CLI structure (Spectre.Console.Cli)

`Program.cs` wires DI (`TypeRegistrar`/`TypeResolver` bridging `Microsoft.Extensions.DependencyInjection` into Spectre) and registers the command tree:

- `connections` (list/add/delete/update) — manage saved metadata-DB connections
- `projects` (list/show/add/update) with sub-branches `sp` (add/remove), `enum` (add/remove), `norm` (add/remove — name normalization rules)
- `generate-code`, `languages-list`, `version`

Conventions to follow when adding commands:
- Commands live under `Commands/<Feature>/`, one class per verb, deriving from `AsyncCommand<Settings>`.
- Settings inherit `GlobalSettings` (adds `--non-interactive`, etc.); language-scoped commands use `LanguageScopedSettings`.
- Every DB-touching command starts with `ToolkitHelper.TryResolveDbHelperAsync(connectionService, name)`, which loads the connection, builds a `ToolkitDbHelper`, and runs `DbInfoValidator.ValidateAsync` (checks DB name == `TigerWrapDb` and API-level compatibility via `ExpectedDbInfo`).
- Report failures with `CliHelper.Fail(ToolkitResponseCode, message, logger)` — it logs, prints red markup, and returns `(int)code` as the process exit code. Prefer the existing `ToolkitResponseCode` values; SP return values are cast directly into this enum.
- Interactive prompts (`CliHelper` has selectors for schemas, databases, enums, language options) must be guarded by `Settings.NonInteractive` → return `CliInteractiveNotAllowed` instead of prompting.

## Connections & secrets

`ConnectionService` persists connections to `connections.json` under `%AppData%\ItTiger.net\TigerWrap` (Windows) or `~/.config/ItTiger.net/TigerWrap` (other). Passwords are **DPAPI-encrypted and Windows-only** (`ProtectedData`, `CurrentUser` scope) — decryption silently yields empty string off-Windows or under a different user. These connections point at the TigerWrap **metadata** DB, not the application database whose procedures are being wrapped (the target DB is chosen per project / via `--database-name`).

## Repo layout

- `ItTiger.TigerWrap.Core/` — connection handling, validation, models, and the generated `ToolkitDbHelper` wrappers (Dapper + `Microsoft.Data.SqlClient`).
- `ItTiger.TigerWrap.Cli/` — the `tiger-wrap` CLI (Spectre.Console, NLog config in `NLog.config`, embedded `Resources.resx`).
- `TigerWrapDb/` — SSDT SQL project; the generator engine. SQL grouped by object type (`Tables/`, `Stored Procedures/`, `Functions/`, `Security/`, `DeploymentScripts/`).
- `ItTiger.TigerWrap.Installer/` — Inno Setup installer.
- `docs/` — user docs (`CLI.md`, `ENUMS.md`, `WRAPPERS.md`, `INSTALL.md`).
